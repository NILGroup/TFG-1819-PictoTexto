\chapter{Metodología de trabajo}
\label{cap4:MetodologiaDeTrabajo}

Hemos decido adoptar una metodología que sigue algunas de las características típicas de las metodologías ágiles, necesitábamos una metodología adaptativa, dado que al comienzo del mismo se desconocía la viabilidad del proyecto, el alcance y el como desarrollarlo.
La principales características de nuestra metodología han sido:

\begin{itemize}

\item Comunicación continua. La comunicación con los tutores y entre nosotros,  ha sido constante desde el principio del proyecto con el fin de garantizar la calidad del producto y asesoramiento en la realización. Por ello hemos tenido dos o tres reuniones presenciales al mes con las tutoras y hemos estado en contacto continuo a través del correo electrónico.
Ademas de Replenishment Meetings, reuniones que surgen de la metodología Kanban en las que se priorizan las tareas que se tienen que realizar. Aunque a diferencia de como sugiere está metodología, estas reuniones se realizaban con las tutoras cada dos o tres semanas en vez de cada semana\footnote{http://giropa832.wpengine.com/kanban/}.
\item Entregas constantes. Para intentar asegurar la correcta resolución del proyecto se han realizado entregas constantes tanto de memoria como de código. Dos o tres veces al mes la memoria y una versión funcional de la aplicación eran revisadas por nuestras tutoras.
\item Se otorga valor al producto entrega tras entrega. Se revisan y se corrigen los errores o posibles modificaciones de entregas anteriores y ademas se añade valor al producto ya sea mediante la adición en la memoria, al código o ambas.
\item Visualización del trabajo en curso. Para ello hemos utilizado un tablero (Figura \ref{fig:ourBoard}) para representar el estado del proyecto en cada momento. Esto además nos ha ayudado a la organización, ya que existían limitaciones como no trabajar en el mismo sitio, ni con el mismo horario y necesitábamos un tablero que nos permitiera de una manera rápida saber el estado del trabajo en cada momento.
\item Pool de tareas. No existen limitaciones de temporales a modo de Sprint, por esta razón tenemos un pool de tareas, en el cual se van añadiendo tareas según avanza el proyecto. En las reuniones se da prioridad a esas tareas y se añaden otras nuevas. 
\end{itemize}

\section{Visualización del trabajo}

En la Figura \ref{fig:ourBoard} se puede ver como esta estructurado el tablero, además de su estado al principio y al final del proyecto. Las tareas en nuestro proyecto pueden ser de dos tipos: memoria o código, y el tablero posee las siguientes columnas:

\begin{itemize}
\item To Do: en esta columna aparecen listadas aquellas tareas que están por realizar. Las tareas están ordenadas por prioridad de manera que arriba siempre aparecen las más prioritarias y que pasaran a ser desarrolladas. Después de cada reunión añadimos nuevas tareas y reajustamos las prioridades.
\item In Progress: Tareas que se están desarrollando. Cada tarea en esta columna debe estar asignada a uno de los miembros del equipo de trabajo. Una misma persona no puede tener más de tarea una asignada, en esta columna, por tanto el WIP(Work In Progress)\footnote{https://kanbanize.com/es/recursos-de-kanban/primeros-pasos/que-es-limite-wip/} de esta columna en nuestro trabajo es dos. 
\item Test: todas las tareas finalizadas por un miembro del equipo pasan a está columna para ser revisadas por el otro. En caso de las tareas de memoria se leen  para ver su corrección y así ademas tener una visión global del proyecto. Para una tarea  de código, se revisa en conjunto para ver que cumpla con las buenas practicas definidas, ver la funcionalidad añadida y hacer test para comprobar la salida del sistema dada una entrada preestablecida, en caso de que sean posible.
\item Revisión: tareas que las tutoras están revisando, para validar su corrección antes de la siguiente reunión. Las tareas de código no son revisadas a nivel interno por las tutoras, tan solo si cumplen con la funcionalidad definida.
\item Done: tareas que tras el proceso de revisión se pueden dar por terminadas.
\end{itemize}

\section{Control de versiones: GitHub}


El sistema elegido para el control de versiones ha sido GitHub, utilizando la cuenta del grupo NYL\footnote{http://nil.fdi.ucm.es/}, utilizar esta cuenta nos permitia tener el proyecto en privado y que solo los miembros del mismo tengan acceso al mismo. Durante el desarrollo del proyecto hemos utilizado diferentes ramas, las cuales explicaremos a continuación:
\begin{itemize}
\item Master: en esta rama se guarda el ultimo estado estable tanto de la aplicación como de la documentación. A final de cada sprint todas las ramas se mergean a master para mantener la actualización y a modo de check-point.
\item Documentation: durante el sprint aquí se realizan todas las modificaciones sobre la documentación del proyecto.
\item Django: esta rama sirve para guardar todas las modificaciones que se hacen en el servidor de Django durante el sprint, probar las nuevas funcionalidades que se están desarrollando etc.
\item Front-End: está rama nació con la necesidad de una aplicación front-end que diera mayor flexibilidad a la aplicación, después de terminar la aplicación sirve para realizar todas las modificaciones sobre dicha aplicación.
\item NLG: en esta rama se ha realizado los diferentes desarrollos sobre el servicio web de generación de lenguaje natural
\item QA: rama temporal para la creación de modificaciones, pruebas y test en las aplicaciones.
\end{itemize}
Hay que destacar que en todas las ramas se mantenía una copia estable del resto del proyecto para realizar las pruebas necesarias y poder tener una visión completa de todo el proyecto durante el desarrollo. 
Antes de cada reunión con las tutoras se hace merge de todas las ramas a la rama master, a modo de check-point.
