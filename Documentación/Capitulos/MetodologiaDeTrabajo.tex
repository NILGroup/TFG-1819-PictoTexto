\chapter{Metodología de trabajo}
\label{cap:MetodologiaDeTrabajo}

Hemos decido adoptar una metodología que sigue algunas de las características típicas de las metodologías ágiles, necesitábamos una metodología adaptativa, dado que al comienzo del mismo se desconocía la viabilidad del proyecto, el alcance y el como lo desarrollarlo.
La principales características de nuestra metodología han sido:

\begin{itemize}

\item Comunicación continua. La comunicación con los tutores y entre nosotros,  ha sido constante desde el principio del proyecto con el fin de garantizar un nivel de calidad en el producto y asesoramiento en la realización. Por ello hemos tenido dos o tres reuniones presenciales al mes con las tutoras y hemos estado en contacto continuo a través del correo electrónico.
Ademas de Replenishment Meetings, reuniones que surgen de la metodología Kanban. Aunque a diferencia de como sugiere está metodología, estas reuniones se realizaban con las tutoras cada dos o tres semanas. Su principal objetivo es establecer prioridades de trabajo\footnote{http://giropa832.wpengine.com/kanban/}.
\item Entregas constantes. Para intentar asegurar la correcta resolución del proyecto se han realizado entregas constantes tanto de memoria como de código. Dos o tres veces al mes la memoria y una versión funcional de la aplicación eran revisadas por nuestras tutoras.
\item Se otorga valor al producto entrega tras entrega. Se revisan y se corrigen los errores o posibles modificaciones de entregas anteriores y ademas se añade valor al producto ya sea mediante la adición en la memoria, al código o ambas.
\item Visualización del trabajo en curso. Para ello hemos utilizado un tablero (Figura ref{fig:ourBoard}) para representar el estado del proyecto en cada momento. Esto además nos ha ayudado a la organización, ya que existían limitaciones como no trabajar en el mismo sitio, ni con el mismo horario, necesitábamos un tablero que nos permitiera de una manera rápida saber el estado actual del trabajo, y en que está trabajando cada miembro.
\item Pool de tareas. No existen limitaciones de temporales a modo de Sprint, por esta razón tenemos un pool de tareas, en el cual se van añadiendo tareas según avanza el proyecto. En las reuniones se dan prioridad a esas tareas y se añaden nuevas. 
\end{itemize}

\section{Visualización del trabajo}

En la Figura \ref{fig:ourBoard} podemos como hemos estructurado nuestro tablero. Las tareas en nuestro proyecto pueden ser de dos tipos: memoria y código. El tablero posee las siguientes columnas:

\begin{itemize}
\item To Do: Tareas por hacer.
\item In Progress: Tareas que se están desarrollando. Cada tarea en esta columna debe estar asignada a uno de los miembros del equipo de trabajo. Una misma persona no puede tener más de tarea una asignada, en esta columna, por tanto el WIP(Work In Progress)\footnote{https://kanbanize.com/es/recursos-de-kanban/primeros-pasos/que-es-limite-wip/} es dos. 
\item Test: todas las tareas finalizadas por un miembro del equipo pasan a está columna para ser revisadas por el otro. En caso de las tareas de memoria se leen  para ver su corrección y así ademas tener una visión global del proyecto. Para una tarea  de código, se revisa en conjunto para ver que cumpla con las buenas practicas definidas, ver la funcionalidad añadida y hacer test para comprobar la salida del sistema dada una entrada preestablecida, en caso de que sean posible.
\item Revisión: tareas que las tutoras están revisando, para validar su corrección antes de la siguiente reunión. Las tareas de código no son revisadas a nivel interno por las tutoras, tan solo si cumplen con la funcionalidad definida.
\item Done: tareas que tras el proceso de revisión se pueden dar por terminadas.
\end{itemize}

\section{Control de versiones: GitHub}


El sistema elegido para el control de versiones ha sido GitHub dado que nos permitía tener el proyecto en privado\footnote{https://help.github.com/articles/making-a-public-repository-private/}, y tener copias de seguridad de todo el proyecto por las diferentes fases por las que el mismo ha pasado.
El uso de un sistema de control de versiones es necesario para mantener estas copias de seguridad y ser accesible para todos los miembros del proyecto desde cualquier sitio. Además nos permite trabajar en paralelo sobre el mismo fichero.
Por ultimo hemos decidido utilizar ramas de funcionalidad\footnote{https://es.atlassian.com/git/tutorials/comparing-workflows/feature-branch-workflow} a nivel de proyecto, tal y como podemos ver en la Figura \ref{fig:ramasGit}.
Antes de cada reunión con las tutoras se hace merge de todas las ramas a la rama master, a modo de check-point.
Esta división en ramas, nos ha permitido poder trabajar en paralelo en las diferentes partes del TFG. En la parte de código esta división nos permitía poder hacer modificaciones y ver la integración de dichas modificaciones con el resto de la aplicación sin necesidad de esperar a que una modificación en cualquiera de las otras aplicaciones supusiera un bloqueo.
